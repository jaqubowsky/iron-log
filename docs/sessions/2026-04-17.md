## Planowanie architektoniczne

Bardzo dużo rozmowy sokratejskiej (~35 min z 1.5h budżetu). Flow:

- **Response shape register** (osobny login vs auto-login) — Jakub początkowo mylił "auto-login" z "rejestracja na cudzy email", myślał że manual login chroni przed atakiem. Thought experiment (atakujący przy manual login vs atakujący przy auto-login + email verification) odblokował. Finalnie pushback Jakuba przeciwko mojej propozycji "auto-login + verified guard" — argument complexity cost bez email service. **Decyzja: register zwraca `{id, email}`, user robi osobny login.**
- **Logout flow** — Jakub chciał `AuthGuard('local')` + `req.logout()`. Oba błędne: local to email+password credentials (logout ich nie ma), req.logout to session-based Passport. Po pytaniach sam doszedł: JwtAuthGuard + revoke refresh token w DB + clear cookie.
- **Refresh token storage** — Paralela do `password` (bcrypt hash w DB). Zgodzona: hash tokena (nie plain), `tokenHash` w DB. Plus decyzja **opaque token** (nie signed JWT) — raw UUID do klienta, SHA256 hash w DB.
- **Hash algorithm** — Entropy reasoning: UUID 128 bit → brute-force unfeasible → SHA256 wystarcza (bcrypt overkill dla high-entropy input, plus 100ms overhead na każdy /refresh). Samodzielnie po jednym pytaniu.
- **Cookies vs body** — Access token w memory (response body), refresh w httpOnly cookie. Argumenty XSS + CSRF + refresh mechanism UX samodzielnie po jednym pytaniu.
- **LocalStrategy signature** — Jakub początkowo `validate(data: ValidateUserInput)` (obiekt). Naprowadzony żeby sprawdzić co passport faktycznie wywołuje. Naprawił na `validate(email, password)` + buduje obiekt wewnątrz strategy (boundary adapter).
- **Separation of concerns** — "strategia powinna dostać już czyste dane" — samodzielnie postawił granicę że LocalStrategy nie wie o passwordHash. Service stripuje.
- **Module structure** — Mój push na `RefreshTokensModule` odrzucony argumentem "auth to cohesive domain, nie rób modułu per tabela używaną tylko tutaj". Zasadny argument, wycofałem się.
