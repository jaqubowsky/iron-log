# Sesja 2026-03-23

## Recall challenge

Pytanie: "Dlaczego explicit M:N (tabela łącząca) zamiast implicit M:N z Prisma? I dlaczego update w $transaction?" (3.5/5) — Pole `order` jako powód explicit M:N — trafnie. "Większa kontrola" — za ogólne, nie sprecyzował nad czym. Transakcja — rozumie atomowość (albo wszystko albo nic), ale nie wymienił konkretnych operacji (delete existing + update with create).

## Co robił

Response transformacja dla WorkoutTemplatesModule. Stworzył WorkoutTemplateResponseDTO z constructorem który spłaszcza zagnieżdżony response z join table na płaski kształt dla frontendu. Podłączył do wszystkich endpointów (GET all, GET one, create, update). Delete zwraca message. Przetestował HTTP callami.

## Samodzielność (1-5)

3 — Sam napisał mapowanie (map z destrukturyzacją), sam znalazł Prisma.WorkoutTemplateGetPayload, sam znalazł buga z brakującym include w create/update. Ale potrzebował dużo naprowadzania na: czym jest Response DTO pattern (nie wiedział jak zacząć), co powinno być w klasie (pola vs constructor), że pola w DTO = kształt docelowy a nie kształt Prisma, jak otypować dane z Prisma z include. 25 minut eksploracji docs i pytań zanim napisał pierwszą linijkę.

## Explain phase

(3.5/5) — Opisał co zrobił i zna trade-offy (class-transformer = boilerplate, map w service = rozrzucona logika). Pomylił Class Validator z Class Transformer. Nie pamiętał kluczowego uzasadnienia swojej decyzji (jedno źródło prawdy). Po naprowadzeniu przypomniał sobie.
