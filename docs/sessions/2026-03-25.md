## Recall challenge

Pytanie: "Jak działa offset pagination pod spodem w SQL? Co się stanie z performance gdy ktoś odpyta page=5000?" (3/5) — Wiedział że SKIP/TAKE to OFFSET/LIMIT. Wiedział że będzie wolno na dużych offsetach, ale nie potrafił powiedzieć DLACZEGO (baza musi przeskanować i odrzucić wszystkie wiersze przed offsetem).

## Co robił

1. ACID i transakcje w raw SQL — BEGIN/COMMIT/ROLLBACK w psql, celowe łamanie constraints w transakcji
2. EXPLAIN ANALYZE — Seq Scan vs Index Scan, dodawanie indeksu i weryfikacja
3. Cursor-based pagination dla workout-templates — implementacja od zera (DTO, service, controller)
4. Bug fixy z code review — @Max(50) na limit w obu pagination DTO, @IsInt() + @Min(1) na order
5. TOCTOU — wyjaśnienie konceptu i zrozumienie że FK constraint to prawdziwa ochrona
6. Aktualizacja roadmapy, uproszczenie skilli (session-end, session-start), usunięcie redundantnego weaknesses.md

## Samodzielność (1-5)

3.5 — Cursor pagination zaimplementowany sprawnie: sam wybrał pattern take+1, sam napisał slice, sam zauważył że totalCount zbędny, sam poprawił controller. Bug fixy szybko i samodzielnie. Potrzebował naprowadzania na: psql basics (kolejność BEGIN/COMMIT, cudzysłowy), ACID Consistency, EXPLAIN ANALYZE interpretacja. Progres vs poprzednie sesje — mniej naprowadzania na implementację, więcej samodzielnych decyzji architektonicznych.

## Explain phase

(3.5/5) — ACID: A, I, D poprawnie, Consistency niepewne (powiedział "zasady odczytu/zapisu" zamiast "z poprawnego stanu do poprawnego stanu"). TOCTOU: zrozumiał koncept ale nazwa niepewna, wiedział że FK chroni. Pominął cursor pagination w pierwszym podejściu do explain — dopiero po dopytaniu. Trade-offy offset vs cursor wytłumaczył trafnie.

## Mock interview

Pominął kluczowy trade-off: cursor nie wymaga totalCount więc filtrowanie prostsze, offset wymaga synchronizacji where między count a findMany — łatwy bug.

# Sesja 2 — 2026-03-25

## Recall challenge

Pytanie: "Wytłumacz EXPLAIN ANALYZE — co to jest, co mówi output, jak decydujesz czy dodać indeks?" (3/5) — Wiedział że to narzędzie do sprawdzania wydajności query i znał flow (sprawdź → dodaj indeks → sprawdź ponownie). Nie użył nazw z outputu (Seq Scan vs Index Scan), nie rozróżnił EXPLAIN (plan) od EXPLAIN ANALYZE (faktyczne wykonanie + realne czasy), nie wspomniał kiedy NIE dodawać indeksu (write overhead).

## Co robił

1. Repository pattern dla ExercisesModule — abstract class ExerciseRepository + PrismaExerciseRepository z implements + useClass w module
2. Oddzielenie od typów Prisma — stworzył własny aplikacyjny typ Exercise, usunął import Prisma z abstract class
3. Rozpoczął WorkoutTemplates repository — zdefiniował abstract class, ale nie dokończył (brak czasu)
4. Dyskusja architektoniczna: abstract class vs prosta klasa, YAGNI, koszt abstrakcji, hexagonal architecture, mappery, Zod vs class-validator
5. Explain-concept: repository vs external API wrapper — różnica odpowiedzialności, Strategy Pattern z useFactory, cross-module communication (exports/imports)
6. Aktualizacja CLAUDE.md — dodanie wyjątku: wizualizacja konceptów kodem jest OK (nauka, nie ghostwriting)

## Samodzielność (1-5)

3.5 — Sam napisał repository, implementację i podłączył w module. Sam zauważył problem z importem Prisma typów w abstract class. Sam poprawił nazewnictwo i return type (Exercise[]). Potrzebował naprowadzania na: provide/useClass w module (początkowo provide: ExercisesService zamiast ExerciseRepository), brak @Injectable(), kto odpowiada za include (service vs repository), co z getNonExistingExerciseIds. Dużo czasu poszło na dyskusje architektoniczne kosztem kodowania.

## Explain phase

(3.5/5) — Dobrze opisał co zrobił i dlaczego abstract class wymaga oddzielenia od typów Prisma. Trafnie porównał z prostą klasą. Słabość: na pytanie "dlaczego nie prosta klasa" dał argument o couplingu, ale sam wcześniej odkrył że prosta klasa też daje separation — nie zintegrował tego w odpowiedzi. Po naprowadzeniu poprawił.
