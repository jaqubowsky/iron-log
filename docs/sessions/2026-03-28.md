## Co robił

Sesja teorii HTTP/REST — 4 tematy głęboko (po feedbacku Jakuba o głębokości vs szerokości):

1. **Idempotentność HTTP metod** — GET, POST, PUT, DELETE, PATCH. Budował definicję stopniowo ("ten sam efekt" → "nie zmienia stanu serwera bardziej niż jednokrotne"). PATCH niuans (nie gwarantowanie idempotentny) — nowa wiedza. Idempotency key — rozumiał koncept (front generuje UUID, backend sprawdza w cache)
2. **Kody statusów** — 201 Created (POST), 200 OK (GET/PUT), 204 No Content (DELETE bez body), 404. Początkowo POST = 200, po hincie poprawił na 201
3. **REST vs GraphQL** — głębokie porównanie. Overfetching od razu trafny. Trade-offy: caching (URL vs POST /graphql), query complexity (depth limiting), wersjonowanie (deprecation vs /v1/). Początkowo argument "popularność" zamiast technicznych powodów — po naprowadzeniu techniczne trade-offy. Potem /explain-concept z pełnym wyjaśnieniem obu technologii, kiedy który, quiz z 3 klientami
4. **URL flow (DNS → TCP → TLS → HTTP → server → response → rendering)** — pełne 7 kroków. TCP/TLS nowe koncepty. Jakub sam eksplorował DevTools Timings — screeny z realnych requestów, pytania o Blocked, DNS Resolution, Sending vs Waiting. Zrozumiał TTFB jako czas serwera. Reverse proxy wyjaśniony analogią (recepcjonista). Keep-alive, HTTP/2 multiplexing, HTTP/3 QUIC — wyjaśnione. Forward secrecy w TLS — nowa wiedza
5. **N+1 problem** — koncept (pętla z query = 1+N), rozwiązanie (IN zamiast pętli). Jak Prisma rozwiązuje (2 queries z IN, nie JOIN). Dlaczego nie JOIN — duplikacja danych w płaskich wierszach (10 logów × 5 ćwiczeń = 50 wierszy z duplikatami vs 60 wierszy bez). Trade-off SQL-level joining vs app-level joining
